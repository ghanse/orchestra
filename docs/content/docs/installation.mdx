Orchestra is a set of [agent skills](https://github.com/ghanse/orchestra/tree/main/skills) that can be installed and used with AI coding assistants.
To use these skills, install orchestra as a plugin using your AI assistant's preferred installation method.

<Tabs items={['Databricks Genie Code', 'Claude Code', 'Other AI tools']}>
<Tab value="Databricks Genie Code">
Clone the orchestra repo into your workspace using a [Git folder](https://docs.databricks.com/aws/en/repos/git-operations-with-repos). Clone it under **`/Workspace/Shared`** (e.g. `/Workspace/Shared/orchestra`) rather than your private `/Workspace/Users/<you>` home — when you later deploy the MCP server, the app's service principal must be able to read the source, and it has no access to private user folders by default. Then copy the `skills/` directory into a user-level skills folder:

```bash
databricks workspace import-dir skills /Users/<your-email>/.assistant/skills
```

To make orchestra available for other workspace users, copy `skills/` into a workspace-level skills folder:

```bash
databricks workspace import-dir skills /Workspace/.assistant/skills
```

Genie Code picks up skills from these directories automatically. Skills fire automatically when their description matches your request. 
To invoke a specific skill, use the `@` prefix (e.g. `@migrate translate the ADF pipelines I exported to /Volumes/main/default/adf_export`).

See the [Databricks Genie Code Skills documentation](https://docs.databricks.com/aws/en/genie-code/skills) for more details.
</Tab>

<Tab value="Claude Code">
Orchestra is packaged as a Claude Code plugin. The plugin manifest lives at [`.claude-plugin/plugin.json`](https://github.com/ghanse/orchestra/blob/main/.claude-plugin/plugin.json).  To install 
orchestra, run the following command from a Claude Code session:

```bash
/plugin marketplace add ghanse/orchestra
/plugin install orchestra
```

You can also copy the skill folders into your local `/.claude/skills` folder:

```bash
cp -R skills/{setup,profile,translate,prepare,migrate} ~/.claude/skills/
```

Once installed, the skills can be invoked using `/orchestra:migrate`, `/orchestra:profile`, etc. Claude Code can also invoke the skills when prompted.  See the [Claude Code Plugins documentation](https://docs.claude.com/en/docs/claude-code/plugins) for more details.
</Tab>

<Tab value="Other AI tools">
Any tool that supports the [Agent Skills](https://agentskills.io/) open standard can install the orchestra skills. The general pattern:

1. Copy each skill folder (`skills/setup`, `skills/profile`, `skills/translate`, `skills/prepare`, `skills/migrate`) into the tool's configured skills directory.
2. Make sure the path contains `SKILL.md` directly,
3. Restart the tool if it caches skill metadata at startup.

<Callout type="info" title="Flattening skills files">
If your tool expects a single Markdown file instead of a directory tree, use the following command to flatten Orchestra's skills files:

```bash
cat skills/*/SKILL.md > orchestra-skills.md
```
</Callout>
</Tab>
</Tabs>

## Setting up the Python environment

Orchestra's skills invoke Python modules that may depend on third-party packages. The `setup` skill provisions an isolated virtual environment with the required dependencies.

Run it **once** after installing the skills, before `profile`, `translate`, `prepare`, or `migrate`. Just ask your agent:

> Set up the orchestra environment

The setup script can also be run directly from the plugin root:

```bash
bash <plugin_dir>/scripts/bootstrap.sh
```

Running the setup process will:

1. Check that `python3`, `pip`, and `venv` are available.
2. Create the virtual environment if it doesn't already exist. When running under Databricks (Genie Code or notebooks, detected via `DATABRICKS_RUNTIME_VERSION`), the venv is created at `/Workspace/Users/<current user>/.migration-skills`; everywhere else it is created at `<plugin_dir>/.venv`.
3. Install the `requirements.txt` dependencies into your virtual environment using `pip`.
4. Write the resolved interpreter path to the marker file `<plugin_dir>/.migration-venv`.

The environment is created once and reused. Re-running the script simply confirms the venv exists and its dependencies are satisfied.

<Callout type="warn" title="Python not found">
If `python3`, `pip`, or the `venv` module are missing, the script will print a warning and exit **without** creating anything.
To install Python in your environment, run one of the following commands:

* **macOS:** `brew install python`
* **Debian/Ubuntu:** `sudo apt-get install python3 python3-venv python3-pip`
* **Windows:** [python.org/downloads](https://www.python.org/downloads/) (enable "Add python.exe to PATH")
</Callout>

After the venv exists, every Python command the skills run uses the interpreter recorded in the marker file, with `src/` on `PYTHONPATH`. Read the interpreter path from `<plugin_dir>/.migration-venv` rather than hardcoding it:

```bash
export PYTHONPATH="<plugin_dir>/src"
PY="$(cat <plugin_dir>/.migration-venv)"
"$PY" -m orchestra.adapter inputs profile
```

The marker file points at `<plugin_dir>/.venv` locally or `/Workspace/Users/<user>/.migration-skills` on Databricks. The agent normally runs these commands for you; they are handy for troubleshooting a `ModuleNotFoundError`.

## Running orchestra as an MCP server

Orchestra's phases are also packaged as [Model Context Protocol](https://modelcontextprotocol.io) tools (in [`src/orchestra/mcp/`](https://github.com/ghanse/orchestra/tree/main/src/orchestra/mcp)) so an agent can invoke them directly instead of shelling out to the CLI. The `setup` skill wires this up automatically based on your environment; you can also do it manually. See [Architecture](/docs/architecture) for how the tool layer maps onto the phases.

<Tabs items={['Claude Code / local agents', 'Databricks Genie Code']}>
<Tab value="Claude Code / local agents">
Install the MCP server stack into the venv and run it over stdio:

```bash
PY="$(cat <plugin_dir>/.migration-venv)"
"$PY" -m pip install "mcp>=1.12" "uvicorn>=0.30" "starlette>=0.40"
PYTHONPATH="<plugin_dir>/src" "$PY" -m orchestra.mcp
```

Register it with your MCP client (use the interpreter path from the marker file for `command`):

```json
{
  "mcpServers": {
    "orchestra": {
      "command": "<plugin_dir>/.venv/bin/python",
      "args": ["-m", "orchestra.mcp"],
      "env": { "PYTHONPATH": "<plugin_dir>/src" }
    }
  }
}
```

<Callout type="info" title="Editable install alternative">
If you prefer an installed package over `PYTHONPATH`, run `pip install -e ".[mcp]"` from the plugin root; then `python -m orchestra.mcp` works without setting `PYTHONPATH`.
</Callout>
</Tab>

<Tab value="Databricks Genie Code">
Genie Code connects to a **hosted** MCP endpoint, so the orchestra tools run as a Databricks App that you add in Genie Code's **Custom MCP server** picker. End to end:

<Steps>
<Step>
### Clone orchestra into a shared workspace folder and install the skills

Follow the **Databricks Genie Code** install steps above to clone the repo and copy `skills/` into your skills folder. Clone into **`/Workspace/Shared`** (e.g. `/Workspace/Shared/orchestra`) — `<plugin_dir>` below refers to that path.

<Callout type="warn" title="Use a shared location, not your user home">
The MCP app's service principal cannot read private `/Workspace/Users/<you>` folders by default, and `deploy.sh` deploys the source from `/Workspace/Shared/<app-name>`. Cloning into `/Workspace/Shared` keeps the repo, the deployed source, and team access all in a location every user and the app's service principal can reach. If your workspace restricts `/Workspace/Shared`, use any other folder all users (and the app SP) can read and pass it via `APP_SOURCE_PATH`.
</Callout>
</Step>

<Step>
### Run the setup skill

Ask your agent to **"set up the orchestra environment"** (or run `bash <plugin_dir>/scripts/bootstrap.sh`). On Databricks the `setup` skill detects the environment and, after creating the venv, runs the app deployment in the next step for you. Run it from a workspace **web terminal** if your Genie session can't shell out to the Databricks CLI.
</Step>

<Step>
### Deploy the MCP server as a Databricks App

```bash
bash <plugin_dir>/app/deploy.sh
```

`deploy.sh` stages a self-contained bundle (the app entrypoint plus a vendored copy of the orchestra source), syncs it to **`/Workspace/Shared/mcp-orchestra`** (a location the app's service principal can read — override with `APP_SOURCE_PATH`), and creates/deploys the **`mcp-orchestra`** app. The script prints the app URL; the MCP endpoint is **`<app-url>/mcp`**.

<Callout type="info" title="App naming and the AI Playground">
The `mcp-` prefix on `mcp-orchestra` makes the app auto-listed in the **AI Playground**. Genie Code's **Custom MCP server** picker selects any Databricks App by name regardless of prefix. Override the name with `APP_NAME` if needed.
</Callout>

<Callout type="warn" title="Deploy from a CLI session">
`databricks apps` deploy commands require a Databricks CLI session (workspace web terminal or a local machine); they are **not** available from serverless notebook Python — the same constraint as `databricks bundle deploy`. If "Error: please specify target" appears, the CLI picked up a stray `databricks.yml` from your working directory; `deploy.sh` already isolates against this, so re-run it as-is.
</Callout>
</Step>

<Step>
### Grant access

- **App access:** grant **Can use** on the `mcp-orchestra` app to the users / service principals that will call it (Apps UI → *Permissions*, or `databricks apps set-permissions`).
- **Data access:** grant the app's own service principal access to the catalogs, schemas, and Unity Catalog volumes the migration reads/writes, plus any SQL warehouse used by the reporting tools (`orchestra_record_results` / `orchestra_install_dashboard`).
</Step>

<Step>
### Add it in Genie Code (Agent mode)

MCP servers are only available in Genie Code **[Agent mode](https://learn.microsoft.com/en-us/azure/databricks/genie-code/use-genie-code#modes)**. To add the app:

1. In the Genie Code panel, click **⚙ Settings**.
2. Under **MCP Servers**, click **+ Add Server**.
3. Choose **Custom MCP server** and select the **`mcp-orchestra`** Databricks App.
4. Click **Save**.

Its `orchestra_*` tools are then available in Agent mode (start with `orchestra_phase_inputs` or `orchestra_migrate`) — no prompt changes needed.

<Callout type="info" title="Genie Code custom MCP requirements">
Databricks requires a custom MCP app to be (1) deployed in the **same workspace**, (2) reachable at **`https://<app-url>/mcp`**, and (3) **stateless**. The orchestra server already satisfies these — it sets `stateless_http=True` and adds CORS. If a browser CORS error appears, set the app's `ORCHESTRA_ALLOWED_ORIGINS` env var to your workspace URL and redeploy. MCP access is also capped at **20 tools** across all servers (orchestra exposes 11); enable/disable specific tools in Genie Code settings. See [Connect Genie Code to MCP servers](https://learn.microsoft.com/en-us/azure/databricks/genie-code/mcp).
</Callout>
</Step>

<Step>
### Verify

Open the health endpoint `<app-url>/` (returns `{"status":"ok"}`), or ask Genie Code **"what orchestra MCP tools are available?"** — you should see the eleven `orchestra_*` tools.
</Step>
</Steps>
</Tab>
</Tabs>

## Verifying the installation

Open your agent and ask:

> What orchestra skills do you have available?

You should see all five skills listed with their descriptions. If only some appear, double-check the install path your tool watches for skills.
