## Motivation

Orchestration should be treated as a first class citizen during migrations. Because orchestrators drive the execution of data pipelines, 
their configuration can impact data processing results as much as the logic being orchestrated. While significant tooling exists for code 
conversion and data reconciliation, migrating from legacy orchestration systems is often manual, time-consuming, and prone to risk.

Orchestra was created to automate the migration of data pipeline configuration in various orchestrators. It provides a robust, tested set 
of capabilities to parse existing data pipeline definitions, create migration artifacts, and convert data pipeline definitions to Databricks' 
[Lakeflow jobs framework](https://docs.databricks.com/aws/en/jobs/).

## How Orchestra works

Orchestra is a set of agent skills and deterministic translators. The translators parse, translate, and generate Databricks resources while 
the skills tell an agentic tool (e.g. Databricks Genie Code, Claude Code, or any agent that supports the open [Agent Skills](https://agentskills.io/) format) 
how to call the deterministic translators.

Translation runs in three phases:

1. `INGEST` parses ADF ARM JSON (e.g. pipelines, datasets, linked services, triggers) into an execution tree and builds a JSON inventory 
that classifies every activity as *deterministic*, *agentic*, or *unsupported*.
2. `TRANSLATE` processes the inventory and converts each activity into a Databricks-compatible intermediate representation. *Deterministic 
activities* are translated by Python handlers while *agentic activities* are handed off to an LLM-assisted translator with the right context. 
*Unsupported activities* are flagged as explicit gaps.
3. `PREPARE` converts each translated pipeline into a Databricks Asset Bundle. Each bundle includes job configuration files, generated notebooks, 
and setup scripts for supporting resources (e.g. Databricks Secret Scopes or Unity Catalog connections).

Each phase can be run independently, maintains its own input/output contract, produces artifacts you can inspect before moving to the next phase.

## Common use cases

Orchestra is built for three scenarios:

1. *Migrating Azure Data Factory pipelines to Lakeflow Jobs* is the primary use case. Orchestra creates deployable job bundles from a collection of ADF pipelines.
2. *Bulk modification of existing Lakeflow Jobs* can be performed programmatically once ADF pipelines have been translated to intermediate representations.
The preparer can build new bundles with updated compute, runtime versions, secret scopes, or notebook paths.
3. *Estimating migration scope* can be done after running the ingest phase. Orchestra will output a strategy breakdown across the ADF export and count
how many activities are deterministic, how many need agentic translation, and how many are unsupported.
